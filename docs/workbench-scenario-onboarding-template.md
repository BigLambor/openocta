# 运维工作台场景接入模板

> 适用范围：新增 Flink、Spark、HDFS、GBase、FI、开发治理、数据 App 等运维场景。  
> 目标：确保每个场景都能进入统一的“技术域上下文 -> 运维对象 -> 场景/专项 -> 工作流闭环”体系。

## 1. 场景基本信息

| 字段 | 填写要求 |
|------|----------|
| 场景 ID | 使用稳定小写 ID，例如 `gbase-lock-wait` |
| 场景名称 | 面向运维人员的名称，例如“GBase 锁等待诊断” |
| 技术域 | `hadoop` / `gbase` / `fi` / `governance` / `dataapps` |
| 工作中心 | 事件、巡检、诊断、治理、容量性能、变更护航之一 |
| 负责人 | 明确团队或角色 |
| 成熟度 | 规划中、Beta、已接入、自动化闭环 |
| 自动化等级 | 手动分析、建议采纳、审批后执行、闭环执行 |

## 2. 对象模型

| 字段 | 填写要求 |
|------|----------|
| 对象类型 | 例如 `cluster`、`flink_job`、`database_instance`、`schedule_chain` |
| 对象实例来源 | 集群列表、作业列表、数据库实例、调度链路或元数据资产 |
| 默认对象范围 | 没有实例时必须能回退到“全部对象” |
| 时间范围 | 至少支持近 1 小时、近 24 小时、近 7 天、近 30 天 |
| 深链参数 | 明确从驾驶舱或域详情跳转时携带的 `domain`、`view`、`scenarioId`、`objectRef` |

## 3. 数据源和输入证据

| 字段 | 填写要求 |
|------|----------|
| 主数据源 | 指标、日志、告警、JMX、调度平台、元数据平台等 |
| 辅助数据源 | 配置、拓扑、历史任务、变更记录、Runbook |
| 数据源健康 | 数据缺失时页面必须显示原因，不显示假结果 |
| 输入证据 | 列出页面和 AI 分析需要展示的证据项 |
| 数据刷新策略 | 实时、滑动窗口、定时巡检、离线分析 |

## 4. 输出和闭环

| 字段 | 填写要求 |
|------|----------|
| 健康信号 | 健康分、风险等级、趋势或异常状态 |
| 风险证据 | 可追溯的指标、日志、事件或配置差异 |
| 建议动作 | 运维人员可采纳、驳回或转任务的动作 |
| 预期收益 | 降低风险、缩短恢复时间、节省资源或避免 SLA 违约 |
| Runbook | 至少绑定一个 SOP 或标记待补 |
| 执行记录 | 必须写入 `scenarioId`、技术域、对象范围、时间范围、输入、输出和操作者 |

## 5. 权限和安全

| 字段 | 填写要求 |
|------|----------|
| 可见权限 | 复用技术域权限，必要时增加场景级权限 |
| 操作权限 | 诊断、巡检、记录闭环、审批执行分别控制 |
| 高风险动作 | Kill session、重启组件、扩容、回滚等必须审批后执行 |
| 审计要求 | 采纳、驳回、审批、执行和回滚都要可追溯 |

## 6. 验收清单

| 验收项 | 要求 |
|--------|------|
| 场景注册 | 已加入 `ui/src/ui/ops/scenario-registry.ts` |
| 场景目录 | 选择对应技术域和工作中心后可见 |
| 全部技术域 | `all` 只展示目录，不误进入单域专项详情 |
| 对象上下文 | 顶部显示技术域、工作中心、对象范围和时间范围 |
| 骨架态 | 数据未接入时明确显示规划中和缺失数据源 |
| AI 入口 | 能基于场景、对象、时间范围生成分析或建议 |
| 执行记录 | 记录闭环后可按场景追溯 |
| 测试 | 注册表、上下文、输出结构相关测试通过 |
