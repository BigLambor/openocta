# 运维工作台场景化重构任务清单

> 版本：v1.0  
> 日期：2026-06-06  
> 范围：运维工作台、驾驶舱域详情跳转、技术域上下文、场景/专项组织方式。  
> 目标：把工作台从“按页面堆专项”重构为“技术域上下文 + 运维对象 + 场景目录 + 工作流闭环”的可扩展架构。

---

## 0. 设计原则

| 原则 | 说明 | 验收方式 |
|------|------|----------|
| 技术域是上下文，不是功能菜单 | BCH、FI、GBase、开发治理、数据 App 只决定数据范围、对象模型和可用场景 | 从任意入口进入工作台时，上下文显示明确且可追踪 |
| 工作台主导航保持稳定 | 事件、巡检、诊断、治理、容量、变更是工作方式，不随专项增加而膨胀 | 新增专项不新增工作台一级/主 Tab |
| 专项能力目录化 | Flink 作业健康度、Spark 作业调优、HDFS 容量调优等作为场景注册，不硬编码到主结构 | 新增场景只需注册元数据和组件映射 |
| 所有场景落到对象 | 场景必须能定位到域、对象类型、对象实例和时间范围 | 场景详情页能展示当前对象上下文 |
| 所有 AI 输出可闭环 | AI 结论必须能沉淀为建议、任务、执行记录、采纳效果或驳回原因 | 执行记录可追溯到场景、对象、输入和输出 |

状态约定：

```text
⬜ 待开始
🟡 进行中
✅ 已完成
⚠️ 阻塞
❌ 不通过
```

## 建议推进顺序

1. 先做 P0，修正上下文和 `all` 语义，避免后续继续叠加错误心智。
2. 再做 P1/P2，完成工作台结构和场景注册表。
3. 用 P3 把现有 BCH 三个专项迁移进去，验证新结构不是纸面设计。
4. P4 开始复制到 GBase/FI/开发治理/数据 App。
5. P5 用于长期运营，重点建设场景效果、生命周期和 Runbook 绑定。
