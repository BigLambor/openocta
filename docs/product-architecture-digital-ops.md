# OpenOcta 智能运维产品骨架与数字员工中心设计方案

> 版本：v0.1  
> 日期：2026-06-02  
> 状态：待产品、架构、研发评审确认  
> 适用范围：OpenOcta 智能运维平台、运维技术域、业务场景、数字员工体系、存量功能改造

---

## 1. 背景与问题

当前平台已经围绕多个运维技术域展开：

- 运维大屏
- BCH 生态 / Hadoop Ecosystem
- FI 商业生态
- GBase 数据库
- 开发治理平台
- 数据 App 运维

同时，每个技术域下开始沉淀具体业务场景，例如：

- 大数据专家 Agent
- 大数据组件告警降噪
- 大数据集群深度巡检
- Flink 作业健康度
- Spark 作业诊断
- HDFS 健康度诊断
- 集群健康度诊断

如果继续把这些场景平铺到菜单或页面中，平台会很快出现以下问题：

1. **点状堆砌**：每新增一个场景就增加一个入口，缺乏稳定归属。
2. **产品骨架不清晰**：用户无法判断一个能力应该从技术域进入，还是从运维能力进入。
3. **跨域能力重复建设**：告警、巡检、诊断、治理、容量等能力会在 BCH、GBase、FI 等技术域内重复出现。
4. **Agent 概念过于技术化**：直接暴露“XX Agent”不利于业务用户理解，也难以体现岗位价值。
5. **后续扩展风险高**：Flink、Spark、HDFS、Kafka、Hive、YARN 等场景持续增加后，菜单会失控。

因此，本方案的核心目标是：**建立一套能够长期承载多技术域、多运维能力、多数字员工、多业务场景的产品骨架**。

---

## 2. 平台定位

建议将 OpenOcta 智能运维方向的产品定位定义为：

> **面向多技术域的智能运维治理平台，通过统一资产、统一观测、统一诊断、统一治理和数字员工协同，提升企业复杂技术体系的稳定性、效率和可运营性。**

该定位包含四层含义：

| 层级 | 含义 |
|------|------|
| 多技术域 | 覆盖 BCH、FI、GBase、开发治理、数据 App 等不同运维对象 |
| 统一运维能力 | 将可观测、告警、巡检、诊断、治理、容量、变更等能力沉淀为平台能力 |
| 数字员工协同 | 将 Agent 产品化为具备岗位职责、技能、工具、任务产出的数字员工 |
| 运维治理闭环 | 不只回答问题，而是形成发现、诊断、处置、治理、复盘、评估的闭环 |

## 3. 设计理念

### 3.1 技术域是对象，不是能力

BCH、FI、GBase、数据 App、开发治理平台，本质是平台管理的运维对象域。

它们不应该承载所有产品结构的组织责任。否则每个技术域都会重复出现告警、巡检、诊断、治理、容量等能力，最终形成多个孤岛。

### 3.2 能力域是骨架，不是菜单装饰

平台应以稳定的运维能力域组织场景：

- 资产与拓扑
- 可观测与告警
- 健康度与巡检
- 故障诊断与应急
- 治理与优化
- 容量、性能与成本
- 变更、配置与合规
- 数字员工

能力域是长期稳定的，业务场景是持续增长的。

### 3.3 业务场景是能力包，不是一级菜单

Flink 作业健康度、Spark 作业诊断、HDFS 健康度诊断、小文件治理、队列拥塞诊断等，都应该作为某个能力域下的场景包。

例如：

| 业务场景 | 推荐归属 |
|----------|----------|
| Flink 作业健康度 | BCH 生态 / 作业治理 |
| Spark 作业诊断 | BCH 生态 / 作业治理 或 故障诊断与应急 |
| HDFS 健康度诊断 | BCH 生态 / 健康度与巡检 |
| 大数据组件告警降噪 | BCH 生态 / 可观测与告警 |
| 大数据集群深度巡检 | BCH 生态 / 健康度与巡检 |

### 3.4 Agent 是技术实现，数字员工是产品表达

不建议在产品层大量暴露“XX Agent”。

更推荐的表达是：

> **Agent 是底层执行引擎，数字员工是对外产品角色。**

用户面对的不是“一个模型”或“一段 Prompt”，而是一个有职责、有边界、有技能、有工具、有任务产出的数字员工。

### 3.5 运维大屏是全局入口，不是技术域

“运维大屏”不应与 BCH、FI、GBase 并列为技术域。它应该是全局驾驶舱，用于聚合所有技术域的健康度、风险、告警、容量、任务和数字员工工作状态。

## 5. 技术域内部标准结构

每个技术域内部建议采用统一能力域结构，避免不同技术域各自发明菜单。

```text
技术域
├── 概览
├── 资产与拓扑
├── 可观测与告警
├── 健康度与巡检
├── 故障诊断与应急
├── 治理与优化
├── 容量性能与成本
├── 变更配置与合规
└── 数字员工
```

### 5.1 概览

展示该技术域的关键状态：

- 健康度评分
- 当前风险
- 未处理告警
- 巡检任务
- 容量水位
- 近期故障
- 数字员工任务状态

### 5.2 资产与拓扑

管理对象、关系、归属、配置和依赖。

典型能力：

- 集群资产
- 组件资产
- 应用资产
- 作业资产
- 服务依赖
- 上下游拓扑
- 责任人和组织归属

### 5.3 可观测与告警

管理指标、日志、事件、链路、告警和异常检测。

典型能力：

- 指标观测
- 日志检索
- 事件流
- 告警聚合
- 告警降噪
- 异常检测
- 告警根因候选

### 5.4 健康度与巡检

管理周期性检查、深度巡检、健康评分和风险发现。

典型能力：

- 日常巡检
- 深度巡检
- 健康评分
- 基线检查
- 风险清单
- 巡检报告
- 趋势对比

### 5.5 故障诊断与应急

管理故障定位、根因分析、影响面分析、处置建议和应急流程。

典型能力：

- 故障诊断
- 根因分析
- 影响面分析
- 应急预案
- 处置建议
- 复盘报告

### 5.6 治理与优化

管理长期问题，而不是只解决一次故障。

典型能力：

- 重复告警治理
- 作业稳定性治理
- 小文件治理
- 资源倾斜治理
- 参数治理
- SLA 治理
- 质量规则治理

### 5.7 容量性能与成本

管理资源水位、容量预测、性能瓶颈和成本归因。

典型能力：

- 容量水位
- 容量预测
- 性能分析
- 资源利用率
- 队列/租户成本
- 成本优化建议

### 5.8 变更配置与合规

管理发布、升级、配置漂移、权限、审计和合规风险。

典型能力：

- 变更前风险评估
- 变更中护航
- 变更后验证
- 配置漂移检测
- 版本兼容检查
- 权限审计

### 5.9 数字员工

展示该技术域内可用数字员工，以及它们的任务、技能、工作成果。

典型能力：

- 领域数字员工列表
- 员工任务入口
- 员工处理记录
- 员工关联技能
- 员工工作成果

## 7. 数字员工中心设计

### 7.1 当前现状判断

当前仓库中已经存在数字员工相关能力：

- `ui/src/ui/views/employee-market.ts`：员工市场，支持搜索、分类、安装、详情、打开、编辑、删除。
- `ui/src/ui/views/digital-employee.ts`：本地数字员工管理，支持创建、Prompt、MCP、Skill、启停、复制、删除和会话入口。
- `src/pkg/employees/model.go`：员工 Manifest 模型，包含名称、描述、Prompt、启用状态、Skill、MCP、来源和类型。
- `src/docs/digital-employees.md`：定义数字员工为垂直场景的专属对话模板。

结论：

> 当前已经具备“数字员工市场 + 本地员工管理 + 会话入口”的基础，但还不是完整的平台级数字员工中心。

当前更像：

```text
数字员工能力现状
├── 找员工
├── 看员工详情
├── 安装员工
├── 创建员工
├── 配置 Prompt / Skill / MCP
└── 开始会话
```

平台级数字员工中心还需要进一步补齐：

- 员工岗位化
- 员工任务化
- 员工编排化
- 员工治理化
- 员工效能化

### 7.3 数字员工角色体系

建议先沉淀 6 类通用岗位，再按技术域实例化。

| 数字员工类型 | 职责 | 对应能力域 |
|--------------|------|------------|
| 值班运维数字员工 | 告警接收、降噪、聚合、初步判断、升级通知 | 可观测与告警 |
| 巡检数字员工 | 日常巡检、深度巡检、健康评分、风险发现 | 健康度与巡检 |
| 诊断数字员工 | 故障定位、根因分析、影响面分析、处置建议 | 故障诊断与应急 |
| 治理数字员工 | 重复问题治理、稳定性治理、配置治理、SLA 治理 | 治理与优化 |
| 容量与成本数字员工 | 容量预测、资源利用率、成本归因、优化建议 | 容量性能与成本 |
| 变更护航数字员工 | 变更前评估、变更中观测、变更后验证、回滚建议 | 变更配置与合规 |

按技术域实例化后，例如：

- BCH 值班运维数字员工
- BCH 深度巡检数字员工
- BCH 作业诊断数字员工
- GBase 数据库诊断数字员工
- FI 组件巡检数字员工
- 数据 App 稳定性数字员工
- 变更护航数字员工

### 7.4 员工档案字段建议

当前 `Manifest` 建议逐步扩展为更完整的员工档案。

建议新增字段：

| 字段 | 说明 |
|------|------|
| `domainKeys` | 服务的技术域，例如 `bch`、`gbase`、`fi` |
| `capabilityKeys` | 归属能力域，例如 `observability`、`inspection`、`diagnosis` |
| `roleType` | 员工岗位类型，例如 `oncall`、`inspector`、`diagnoser` |
| `responsibilities` | 岗位职责列表 |
| `inputSources` | 输入来源，例如告警、指标、日志、工单、巡检任务 |
| `outputTypes` | 输出成果，例如诊断报告、巡检报告、治理任务 |
| `actionScopes` | 可执行动作范围，例如只读、创建工单、执行脚本 |
| `permissionKeys` | 权限点 |
| `runbookRefs` | 关联 Runbook |
| `knowledgeRefs` | 关联知识库 |
| `evaluationMetrics` | 员工效能指标配置 |

这部分不要求一次性全部实现，可以先在前端和文档中确立概念，再分阶段落地。

## 8. 运维能力中心设计

运维能力中心用于跨技术域沉淀通用能力，避免能力重复建设。

### 8.1 能力中心与技术域的关系

```text
技术域运维：从对象进入
能力中心：从能力进入
数字员工中心：从岗位进入
```

三种入口面向不同使用心智：

| 入口 | 用户问题 |
|------|----------|
| 技术域运维 | 我想看 BCH / GBase / FI 的情况 |
| 运维能力中心 | 我想处理告警 / 巡检 / 诊断 / 治理 |
| 数字员工中心 | 我想找一个员工帮我完成工作 |

### 8.2 能力中心建议模块

| 能力中心 | 作用 |
|----------|------|
| 资产与拓扑中心 | 跨域统一资产、依赖、拓扑、责任人 |
| 可观测中心 | 跨域指标、日志、事件、链路 |
| 告警事件中心 | 告警接入、降噪、聚合、事件管理 |
| 巡检中心 | 巡检模板、巡检计划、巡检报告 |
| 诊断中心 | 诊断任务、根因分析、影响面分析 |
| 治理中心 | 长期问题、治理任务、治理效果 |
| 容量性能中心 | 容量预测、性能瓶颈、成本分析 |

## 9. 存量改造方案

### 9.1 菜单层改造

当前菜单中已经有运维大屏、技术域、数字员工、员工市场等入口。建议改造为：

其中：

- 原 BCH、FI、GBase、开发治理、数据 App 归入“技术域运维”。
- 原告警降噪、巡检、诊断等通用能力逐步归入“运维能力中心”。
- 原 `employee-market` 和 `digital-employee` 归入“数字员工中心”。
- 原 Skills、MCP、模型、权限、通道、安全策略归入“系统管理”。

### 9.2 `employee-market` 改造

当前 `employee-market` 保留，但重新定位为：

> **数字员工中心 / 员工市场**

建议增强：

1. 增加按技术域筛选：BCH、GBase、FI、数据 App、开发治理。
2. 增加按岗位类型筛选：值班、巡检、诊断、治理、容量、变更。

4. 员工详情中展示岗位职责、输入来源、输出成果、权限边界。
5. 安装后进入“我的员工”，而不只是开始会话。

### 9.3 `digital-employee` 改造

当前 `digital-employee` 保留，但重新定位为：

> **数字员工中心 / 我的员工**

建议增强：

1. 从“对话模板列表”升级为“员工资产管理”。
2. 增加员工档案页。
3. 增加员工任务记录。
4. 增加员工关联技术域和能力域。
5. 增加员工权限、工具调用、执行动作配置。
6. 增加员工效能指标。

### 9.4 `agent-swarm` 改造

当前 `agent-swarm` 可作为：

> **数字员工中心 / 员工编排**

建议弱化“蜂群”技术感，强化“协同工作”产品表达。

推荐命名：

- 员工协同
- 数字员工编排
- 多员工工作间

适用场景：

- 告警员工发现问题后转派诊断员工。
- 巡检员工输出风险后转派治理员工。
- 变更护航员工调用观测员工验证变更结果。
- 主员工拆分任务给多个子员工执行。

### 9.5 告警能力改造

原“告警降噪与评估”建议拆成两个视角：

| 视角 | 归属 |
|------|------|

| 技术域内告警 | 技术域 / 可观测与告警 |

这样既能从全局处理告警，也能在 BCH、GBase 等技术域内查看本域告警。

### 9.6 巡检能力改造

原“深度巡检”建议拆成：

| 视角 | 归属 |
|------|------|
| 巡检中心 | 运维能力中心 |
| 技术域巡检 | 技术域 / 健康度与巡检 |
| 巡检数字员工 | 数字员工中心 |

巡检不再只是一次 Agent 对话，而是具备模板、计划、任务、报告、趋势和风险闭环。

## 15. 验收口径

本方案落地是否成功，不以页面数量衡量，而以以下标准衡量：

1. 新增一个业务场景时，能在 1 分钟内判断它属于哪个技术域、哪个能力域、哪个数字员工。
2. 新增 Flink、Spark、HDFS、Kafka、Hive 等场景时，不需要新增一级菜单。
3. 用户能从技术域、能力中心、数字员工三个视角进入同一个工作闭环。
4. 数字员工不只是聊天入口，而能形成任务、报告、记录、审计和效能。
5. 告警、巡检、诊断、治理等能力能够跨技术域复用。
6. 产品结构能支撑未来 1-2 年的场景扩展，而不需要频繁推倒重来。

## 17. 总结

本方案的核心判断是：

> **技术域是入口，能力域是骨架，业务场景是能力包，数字员工是执行层，任务闭环是价值证明。**

OpenOcta 智能运维方向不应继续沿着“每个场景一个入口”的方式扩展。更稳妥的产品路线是：

1. 用技术域承载运维对象。
2. 用能力域组织运维行为。
3. 用场景包承载具体业务场景。
4. 用数字员工产品化 Agent 能力。
5. 用任务、报告、审计、效能形成运营闭环。

这套结构会直接影响产品后续的可扩展性、用户理解成本和商业表达能力，应作为后续运维产品建设的基础骨架。
