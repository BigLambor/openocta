# OpenOcta 智能运维架构评审意见

> 版本：v0.1  
> 日期：2026-06-04  
> 目的：对 `product-architecture-digital-ops.md` 与 `architecture-paradigm-analysis.md` 的关键分歧进行评审，重点判断“数字员工”方案是否适合作为 OpenOcta 的产品主轴。

---

## 1. 总体结论

我的判断是：

> **专家建议总体合理，尤其是对“数字员工中心”作为一级产品主轴的质疑，建议采纳。**

但这不等于完全否定“数字员工”概念。更准确的调整方向是：

> **保留数字员工作为 AI 场景助手、自动化模板或垂直任务角色，但不建议把数字员工中心建设成平台一级骨架。**

OpenOcta 更适合演进为：

```text
结构化运维平台
+ 全局 AI 助手
+ 场景内 AI 辅助
+ 自动化 / Runbook / Skill / MCP 引擎
```

而不是：

```text
数字员工管理平台
```

也就是说，平台的主心智应该是“看状态、处理事件、做巡检、诊断故障、治理问题、护航变更”，不是“找员工、管理员工、评价员工”。

---

## 2. 原方案中值得保留的判断

`product-architecture-digital-ops.md` 中有几条判断是正确的，建议继续保留。

### 2.1 技术域是对象，不是能力

BCH、FI、GBase、开发治理、数据 App 本质是被管理的运维对象域。它们不应该各自重复建设告警、巡检、诊断、治理、容量、变更等完整能力。

这个判断是对的。否则每个技术域都会形成一套局部运维系统，最终出现能力重复、命名不一致、数据孤岛和治理割裂。

### 2.2 能力域是长期骨架

资产与拓扑、可观测与告警、健康度与巡检、故障诊断与应急、治理与优化、容量性能与成本、变更配置与合规，这些能力域是稳定的。

业务场景会增长，但能力域不应频繁变化。用能力域承载场景包，比把 Flink、Spark、HDFS、小文件治理、队列拥塞诊断等场景平铺为菜单更合理。

### 2.3 业务场景是能力包，不是一级菜单

Flink 作业健康度、Spark 作业诊断、HDFS 健康度、小文件治理等，应该挂在稳定能力或工作流下面，而不是每新增一个场景就新增一个一级入口。

这个方向应继续坚持。

## 3. 专家建议中我认为合理的部分

### 3.1 不应回退到纯对话框

专家并没有主张 OpenOcta 采用“一个输入框解决一切”的纯 AI-Native Chat 模式。相反，他明确承认企业运维需要结构化骨架。

这一点很重要。

企业运维平台需要：

- 全局健康态势
- 告警与事件生命周期
- 巡检计划和巡检报告
- 变更前、中、后流程
- 治理任务闭环
- 权限、审计、审批、追溯
- 团队协同和任务状态

这些不是一个对话框可以独立承载的。

因此，OpenOcta 不应选择“纯对话框”，而应选择“结构化平台 + AI 增强”。

### 3.2 三个平行入口会制造认知负担

原方案中同时存在：

- 技术域运维
- 运维能力中心
- 数字员工中心

这三个入口都可能进入相同或相近的能力。例如告警降噪既可能在 BCH 技术域内，也可能在告警事件中心，也可能由值班运维数字员工承载。

这会导致用户产生入口选择问题：

> 我到底应该从 BCH 进去，从告警中心进去，还是从数字员工进去？

企业运维用户不希望先理解产品的信息架构。他们希望直接完成工作：

- 当前哪些系统有风险？
- 哪些告警需要处理？
- 这次故障根因是什么？
- 巡检发现了什么？
- 变更是否安全？
- 哪些问题需要治理？

所以专家提出“减少入口、弱化多路径”的建议是合理的。

### 3.3 AI 不应被限制在“数字员工中心”

原方案把 Agent 产品化为数字员工，并进一步建设员工市场、我的员工、员工档案、员工工作台、员工编排、权限审计、效能评估。

这会让 AI 能力被封装在一个独立中心里，容易形成两个问题：

1. 用户要先找到某个员工，才能使用 AI 能力。
2. 页面内的实际工作流反而缺少即时 AI 辅助。

更合理的方式是：

- 在告警详情页直接提供“分析根因”
- 在巡检报告页直接提供“解释风险”
- 在作业失败页直接提供“诊断失败原因”
- 在变更单页直接提供“评估变更风险”
- 在治理任务页直接提供“生成治理建议”

AI 应该出现在用户工作发生的位置，而不是要求用户离开当前上下文去“找员工”。

### 3.4 “数字员工效能”容易变成虚假指标

自动闭环率、诊断采纳率、节省人时、告警降噪率等指标本身可以保留，但不建议包装成“员工绩效”。

原因是：

- 初期样本少，指标可信度低。
- 很多采纳、节省人时需要人工反馈，数据容易失真。
- “员工绩效”会强化拟人化心智，抬高用户预期。

更合适的命名是：

> **自动化效果看板**

它评估的是平台能力和自动化流程，而不是评估一个“员工”。

## 4. 我对“数字员工”的判断

### 4.1 数字员工不是完全错误

“数字员工”有一定合理性，尤其是在 OpenOcta 当前已有员工市场、本地员工管理、Prompt、Skill、MCP 绑定能力的情况下，完全删除这个概念并不现实，也未必必要。

它的合理价值包括：

- 比 Agent、Prompt、MCP、Skill 更容易被非技术用户理解。
- 可以承载垂直场景的预设能力包。
- 可以把提示词、工具、知识库、权限、输出模板组合成一个可复用对象。
- 适合做面向业务场景的助手模板，例如 BCH 巡检助手、Spark 诊断助手、变更护航助手。

因此，我不建议简单说“数字员工不能做”。

### 4.2 数字员工不适合作为一级产品主轴

问题不在于“数字员工”这个词本身，而在于它是否承担了过多产品结构责任。

如果数字员工承担以下职责，就会过重：

- 一级导航入口
- 独立中心
- 员工档案
- 员工工作台
- 员工编排
- 员工效能
- 员工治理
- 技术域内员工列表
- 员工市场作为主要能力发现方式

这会让平台从“运维工作平台”偏向“数字员工管理平台”。

对运维人员来说，他们的核心任务不是管理员工，而是解决告警、故障、巡检、变更、容量和治理问题。

### 4.3 建议重新定义数字员工

建议将数字员工重新定义为：

> **面向特定运维场景的 AI 助手模板，封装 Prompt、Skill、MCP、知识、权限和输出格式，用于在具体工作流中辅助用户完成任务。**

这个定义有几个关键点：

- 它是助手模板，不是真员工。
- 它服务于场景，不独立于场景。
- 它可以被全局 AI 助手调用，也可以被页面内按钮触发。
- 它可以参与自动化流程，但不要求用户管理一套员工组织。

可选命名：

| 原命名 | 建议命名 |
|--------|----------|
| 数字员工中心 | AI 助手库 / 场景助手库 / 自动化模板库 |
| 员工市场 | 助手模板市场 / Skill 模板市场 |
| 我的员工 | 我的助手 / 已安装助手 |
| 员工档案 | 助手配置 / 场景配置 |
| 员工编排 | 自动化工作流编排 |
| 员工效能 | 自动化效果 |

如果出于市场表达仍希望保留“数字员工”，建议只在营销或解决方案层使用，不要在核心产品导航里过度展开。

## 5. 建议的目标架构

建议 OpenOcta 采用以下产品骨架。

```text
OpenOcta 智能运维平台
├── 运维驾驶舱
│   ├── 全局健康态势
│   ├── 技术域健康概览
│   ├── 告警与事件概览
│   ├── 巡检与风险概览
│   ├── 容量与成本概览
│   └── 自动化执行概览
│
├── 服务与资产
│   ├── 技术域资产
│   ├── 集群 / 实例 / 组件 / 作业
│   ├── 服务依赖
│   ├── 拓扑关系
│   └── 责任人与组织归属
│
├── 运维工作台
│   ├── 事件中心
│   │   └── 告警 → 事件 → 诊断 → 处置 → 复盘
│   ├── 巡检中心
│   │   └── 模板 → 计划 → 执行 → 报告 → 风险
│   ├── 诊断中心
│   │   └── 问题 → 证据 → 根因 → 建议 → 任务
│   ├── 治理中心
│   │   └── 发现 → 分派 → 跟踪 → 验收 → 评估
│   ├── 容量性能中心
│   └── 变更护航中心
│
├── AI 与自动化
│   ├── 全局 AI 助手
│   ├── 场景助手模板
│   ├── Skill / MCP / Runbook
│   ├── 自动化工作流
│   ├── 触发规则
│   ├── 审批策略
│   └── 执行记录与审计
│
└── 系统管理
    ├── 模型配置
    ├── 权限管理
    ├── 安全策略
    ├── 通道配置
    └── 平台配置
```

这个架构的核心差异是：

- 技术域不再承担大量二级能力菜单，而是更多承担资产、概览和上下文。
- 运维工作台承载主要工作流。
- AI 与自动化作为增强层和配置层，不作为用户解决问题的主入口。
- 数字员工降级为场景助手模板，服务于页面和工作流。

## 6. 推荐交互模式

### 6.1 全局 AI 助手

用户在任何页面都可以唤起全局 AI 助手。

全局 AI 助手应自动注入当前上下文：

- 当前技术域
- 当前资产
- 当前告警 / 事件 / 巡检 / 变更
- 当前用户权限
- 可调用工具范围

它的输出不应只是一段回答，还应该能：

- 跳转到相关页面
- 生成诊断报告
- 创建治理任务
- 触发 Runbook
- 请求审批
- 生成复盘草稿

### 6.2 场景内 AI 辅助

每个关键页面都应该有上下文相关的 AI 操作。

示例：

| 场景 | AI 操作 |
|------|---------|
| 告警详情 | 分析根因、聚合相似告警、判断影响面 |
| 事件详情 | 生成处置建议、推荐 Runbook、生成复盘 |
| 巡检报告 | 解释风险、生成治理任务、生成趋势总结 |
| 作业失败 | 诊断失败原因、关联日志和指标、推荐修复方案 |
| 变更单 | 评估风险、生成观测项、变更后验证 |
| 容量页面 | 预测水位、识别热点、生成扩容或优化建议 |

### 6.3 自动化触发

AI 不应只靠用户对话触发，也应该能被事件和规则触发。

例如：

- 高等级告警触发根因分析。
- 每日定时触发巡检。
- 变更开始前触发风险评估。
- 巡检发现风险后自动生成治理任务。
- 告警恢复后生成事件总结。

这部分应归入自动化工作流，而不是“员工编排”。

## 7. 对现有数字员工能力的改造建议

当前仓库已有员工市场、本地数字员工管理、Manifest、Prompt、Skill、MCP 绑定能力。建议不要推倒重来，而是渐进改造。

### 7.1 员工市场

建议从“员工市场”逐步改为：

> **助手模板市场** 或 **场景助手库**

保留安装、搜索、分类、详情能力，但分类建议改为：

- 技术域：BCH、GBase、FI、数据 App、开发治理
- 工作流：告警、巡检、诊断、治理、容量、变更
- 工具能力：日志分析、指标分析、SQL、Runbook、MCP

### 7.2 我的员工

建议从“我的员工”逐步改为：

> **已安装助手** 或 **助手配置**

保留 Prompt、Skill、MCP 配置，但不要扩展过重的岗位档案。

建议字段控制在：

- 名称
- 描述
- 适用场景
- 技术域
- 工作流类型
- Prompt
- Skill / MCP
- 权限范围
- 输出模板
- 启用状态

### 7.3 员工编排

建议改为：

> **自动化工作流编排**

编排对象不应是“多个员工协同”，而应是：

- 触发条件
- 输入数据
- 执行步骤
- 调用工具
- 审批节点
- 输出结果
- 执行记录

### 7.4 员工效能

建议改为：

> **自动化效果**

可保留指标：

- 执行次数
- 成功率
- 平均耗时
- 人工采纳率
- 自动闭环率
- 降噪率
- 成本消耗

但不要表达为“员工绩效”。

## 8. 落地优先级建议

### P0：先验证一个闭环场景

不要先铺完整架构。建议选择 BCH 作为样板，但只做一个高价值闭环：

```text
告警进入
→ 告警聚合 / 降噪
→ 事件生成
→ AI 根因分析
→ 处置建议 / Runbook 推荐
→ 人工确认或自动执行
→ 结果记录
→ 复盘草稿
```

这个闭环能跑通，比先建设完整数字员工中心更有价值。

### P1：重构导航和入口心智

建议先收敛一级导航：

- 运维驾驶舱
- 服务与资产
- 运维工作台
- AI 与自动化
- 系统管理

如果暂时不能改这么大，也至少不要继续扩张“数字员工中心”的子模块。

### P2：把 AI 嵌入关键场景

优先在以下页面加上下文 AI 能力：

- 告警详情
- 事件详情
- 巡检报告
- 作业诊断
- 变更护航

### P3：沉淀助手模板，而不是员工组织

等 1-2 个场景跑通后，再把有效 Prompt、Skill、MCP、输出格式沉淀为助手模板。

此时“数字员工”如果要保留，也应作为模板的别名或业务包装，而不是新增管理体系。

### P4：建设自动化效果看板

当自动化执行记录足够多，再建设效果看板。

初期不建议承诺“节省多少人时”这类强价值指标，优先展示可验证指标：

- 执行次数
- 成功 / 失败
- 失败原因
- 平均耗时
- 人工确认次数
- 自动创建任务数

## 9. 需要评审的问题

建议后续评审重点围绕以下问题展开：

1. OpenOcta 的一级用户心智到底是“运维工作平台”，还是“数字员工平台”？
2. 数字员工是否必须保留为产品命名？如果保留，是营销概念、模板概念，还是导航概念？
3. 技术域运维和运维能力中心是否会形成重复入口？是否需要合并为“服务与资产 + 运维工作台”？
4. BCH 样板场景优先做“告警闭环”，还是“巡检闭环”，还是“作业诊断闭环”？
5. 当前员工市场、digital-employee、agent-swarm 的既有实现，是继续强化“员工”，还是改造成“助手模板 + 自动化工作流”？
6. 平台第一阶段最能证明 AI 价值的可观测指标是什么？

---

## 10. 最终建议

我的最终建议是：

> **采纳专家对数字员工体系过重的批评，但不要简单删除现有数字员工能力。应将数字员工降级为“场景助手模板”，把平台主骨架调整为“驾驶舱 + 服务资产 + 运维工作台 + AI 自动化”。**

这样既能保留 OpenOcta 已有的 Agent、Skill、MCP、员工市场能力，又能避免产品心智偏向“管理员工”，从而更贴近企业运维用户真正的工作流。

一句话总结：

> **数字员工可以是能力包装，但不应是平台骨架；OpenOcta 的主线应该是运维闭环，而不是员工管理。**
