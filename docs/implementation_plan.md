# OpenOcta 二次改造 30天实施计划与工作量规划

为了将开源 OpenOcta 深度适配到您的业务中，本方案从**UI/CSS布局改造**、**菜单重构（适配大数据与数据库生态）**、**RBAC 权限系统开发**以及**两个核心 Agent 运维场景（深度巡检、告警降噪与评估）**出发，设计了一份为期 30 天（6月1日 - 6月30日）的详细落地计划与工作量估算。

---

## 架构与技术选型

1. **RBAC 存储**：复用 OpenOcta 原生已引入的 `modernc.org/sqlite`（无 CGO 依赖的纯 Go SQLite 驱动），在 OpenOcta 状态目录下新建 `rbac.db` 或直接在主数据库中新增 RBAC 数据表（`users`, `roles`, `permissions`, `role_permissions`），确保单二进制部署的轻量性与零外部依赖。
2. **安全中间件**：在后端 Go Gateway (`src/pkg/gateway/http`) 中增加 JWT 或 Session 认证拦截器，对除了登录和 Webhook 以外的所有 API 进行权限控制。
3. **前端菜单**：在 `ui/src/ui/navigation.ts` 中重构 Tab 结构，采用角色权限动态过滤机制。
4. **运维场景 1（深度巡检）**：使用内置的 `Cron` 引擎 + 自定义 **MCP Tools / Skills**（编写 Go / TS 抓取 Hadoop JMX、FI 接口及 GBase SQL 指标）配合 LLM 生成巡检报告。
5. **运维场景 2（告警评估与降噪）**：扩展 `/hooks/alert` 接收端，实现内存/SQLite 滑动窗口合并规则，并通过专门的 Agent 提示词评估故障传播链与降噪策略。

## 30 天项目阶段规划

### 阶段一：UI 重构与菜单定制（6月1日 - 6月7日）
* **目标**：实现符合您团队职责划分的前端菜单和高级运维视觉设计。
* **主要工作**：
  - **CSS 视觉优化**：微调布局样式，调整为符合现代数据运维系统的高质感暗黑/科技感主题，加入平滑过渡微动画。
  - **菜单重构**：修改 `ui/src/ui/navigation.ts`，将原来通用的 Agent 菜单重构成以下模块：
    1. **运维大屏 (Overview)**：综合集群状态与告警图表。
    2. **Hadoop 生态运维 (Hadoop Ops)**：开源 Hadoop、HDFS、YARN、Spark 的交互式 Agent 巡检与执行。
    3. **FI 商业生态运维 (FusionInsight Ops)**：商业闭源生态的状态跟踪、诊断和指令下发。
    4. **GBase 数据库运维 (GBase Ops)**：GBase 数据库的性能、死锁、慢 SQL 智能分析。
    5. **开发治理平台运维 (Governance Ops)**：元数据、数据质量、数据血缘巡检。
    6. **数据 App 运维 (Data Apps Ops)**：30 多个数据 App（数仓任务、BI 报表）的链路与调度健康度。
    7. **告警降噪与评估 (Alert Studio)**：查看合并告警组、影响树状图和降噪报告。
    8. **系统设置 (System)**：包含原本的技能库、MCP 工具、模型库以及新增的“权限管理”。
  - **前端路由与占位**：为上述每个运维菜单创建基础页面骨架 (`ui/src/ui/views/`)。

### 阶段二：RBAC 权限系统开发（6月8日 - 6月14日）
* **目标**：为产品引入多租户/多角色安全隔离机制。
* **主要工作**：
  - **后端数据模型**：
    - 在 SQLite 中创建用户表、角色表、权限点表和角色权限关系表。
    - 预置角色：`admin` (超级管理员)、`operator` (日常运维人员，仅能操作自己模块)、`viewer` (只读访客)。
  - **后端 API 校验**：
    - 开发注册、登录（JWT 签发）API。
    - 实现 Go 中间件对 `src/pkg/gateway` 中的 API 进行拦截过滤。
  - **前端权限管控**：
    - 接入登录页面，保存 Token。
    - 在 `navigation.ts` 中实现基于角色权限（Permissions）的菜单过滤展示。
    - 前端组件层对敏感按钮（如：修改配置、重置任务）进行权限置灰或隐藏。
  - **权限管理页面**：在前端“系统设置”下提供用户列表、角色分配、权限点勾选界面。

### 阶段三：深度巡检场景开发（6月15日 - 6月21日）
* **目标**：实现对 Hadoop、FI、GBase 的自动化指标抓取和 Agent 深度巡检。
* **主要工作**：
  - **巡检工具开发 (MCP/Skills)**：
    - **Hadoop MCP**：对接 HDFS NameNode/ResourceManager JMX，采集 JVM 状态、节点死活、队列使用率。
    - **FI MCP**：对接 FusionInsight Manager REST API，采集核心告警事件及组件健康状态。
    - **GBase MCP**：调用 GBase CLI/API 执行 SQL，获取当前活跃会话、锁冲突及表空间余量。
  - **定时巡检流控制 (Cron)**：
    - 在内置 `Cron` 引擎中注册定时任务（例如：每天凌晨及下午 5 点各执行一次）。
    - 任务触发时，由 Agent 调用上述 MCP 工具链搜集各项原始指标。
  - **巡检 Prompt 优化与报告生成**：
    - 设计结构化巡检 Prompt，指导 LLM 从指标数据中提炼风险点、容量瓶颈并给出处理建议。
    - 前端实现“巡检报告”历史归档页，提供报告 PDF 导出及状态趋势图表。

### 阶段四：告警降噪与影响分析场景及集成测试（6月22日 - 6月30日）
* **目标**：接入告警 Webhook，实现告警合并与大模型深度影响分析。
* **主要工作**：
  - **告警降噪服务 (Backend)**：
    - 扩展 `/hooks/alert`，实现告警事件缓冲队列。
    - 设计时间窗口合并策略（如 5 分钟内对同一主机、同一业务组、或具有上下游拓扑关联的告警进行合并打包）。
  - **告警分析 Prompt 设计**：
    - 设计“影响评估 Prompt”，让 LLM 基于合并后的告警数据，通过 MCP 查询实时网络/磁盘/应用拓扑，推导“根本原因 (Root Cause)”，给出“受灾范围评估 (Impact Analysis)”以及“降噪屏蔽建议 (Noise Suppression)”。
  - **前端告警评估面板**：
    - 在前端开发“告警评估与降噪”视图，显示当前发生的告警事件、降噪合并比、以及 LLM 的诊断过程。
  - **联调与交付部署**：
    - 与您的监控源（如 Alertmanager, 商业版告警中心）进行 Webhook 对接联调。
    - 整体打包、打包容器镜像，进行性能和权限边界渗透测试，于 6 月 30 日前交付上线。

## 验证与验收标准

### 1. RBAC 系统校验
- 匿名用户无法访问除公开页面（如登录页）之外的任何 API，接口返回 401。
- 登录后根据角色（例如 GBase 运维员）仅能访问 GBase 菜单和执行相关操作；不可查看 FI 和 Hadoop 菜单（前端隐藏，后端接口拦截）。

### 2. 深度巡检效果
- 设置 Cron 任务，在触发时能够通过 MCP 从对应大数据/数据库平台拉取真实状态数据。
- LLM 能自动输出一份包含组件健康状态、警告隐患、扩容建议的中文巡检报告。

### 3. 告警评估与降噪
- 模拟发送 10 条短时间内重复的 YARN 节点告警，系统能将其自动降噪合并为 1 条告警组。
- LLM 的评估报告能准确指出是由哪一个物理节点挂掉引发的次生告警，并给出首要排查建议。
